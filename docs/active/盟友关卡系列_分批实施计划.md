# 盟友关卡系列 — 分批实施计划

> 本文档将 [盟友关卡系列_详细配置文档.md](盟友关卡系列_详细配置文档.md) 中的内容拆分为 **7 个独立批次**。
> 每批完成后可独立测试、独立提交，前后批次有明确依赖关系。
> 每个代码改动都标注了**精确文件、行号、旧代码→新代码**。

---

## 批次总览

| 批次 | 内容 | 代码改动 | 产出 | 可测试点 |
|------|------|---------|------|---------|
| **1** | 机制1：AI初始行为配置 | 3文件~25行 | 代码基础 | set_initial_behavior 生效 |
| **2** | 关卡1：初次并肩 | 0代码 | map17 | 联合作战入门全流程 |

## 外部调用：设置 AI 队友的初始行为（用于分兵驻防）
func set_initial_behavior(behavior: String, pos: Vector2 = Vector2.ZERO) -> void:
	match behavior:
		"defend":
			ally_state = AllyState.FORCE_DEFEND
			_force_target_pos = pos
			unit.move_to(pos)
		"attack_move":
			ally_state = AllyState.FORCE_ATTACK
			_force_target_pos = pos
			unit.attack_move_to(pos)
		"follow", _:
			pass  # 默认 FOLLOW_PLAYER，无需操作
```

func _on_capture_point_captured(_team: int, point: CapturePoint) -> void:
	var delay: float = point.ally_reinforcement_delay
	if delay <= 0.0 or point.ally_reinforcement_groups.is_empty():
		return
	if RelayManager.is_online:
		return  # 多人模式不支持 AI 队友
	var spawn_pos: Vector2 = point.global_position
	var target: Vector2 = _find_nearest_enemy_pos(spawn_pos)
	# 屏幕提示
	spawner_module.show_floating_text(tr("ALLY_REINFORCEMENT_INCOMING"), Color(1.0, 0.85, 0.0), spawn_pos)
	# 第一次延迟增援
	var timer := get_tree().create_timer(delay)
	timer.timeout.connect(_on_ally_reinforcement_timeout.bind(point.ally_reinforcement_groups, spawn_pos, target))
	# === 连续增援波次 ===
	var repeat: int = point.ally_reinforcement_repeat
	if repeat > 0:
		var interval: float = point.ally_reinforcement_interval
		for i in range(repeat):
			var wave_delay: float = delay + (i + 1) * interval
			var rtimer := get_tree().create_timer(wave_delay)
			rtimer.timeout.connect(_on_ally_reinforcement_timeout.bind(point.ally_reinforcement_groups, spawn_pos, target))
```

## 对指定小队下达攻击指令
func issue_squad_attack_order(squad_id: String, world_pos: Vector2) -> void:
	for u in get_tree().get_nodes_in_group("player_units"):
		if not (u is CharacterBody2D):
			continue
		if u.is_dead():
			continue
		if u.owner_id != -2:
			continue
		var ai := u.get_node_or_null("AllyAI")
		if ai != null and ai.squad_id == squad_id:
			ai.issue_attack_order(world_pos)

## 附录：关键风险和注意事项

### 风险1：WaveManager 字段名不确定
关卡2和关卡4的波次配置字段名（`delay`、`spawns`、`attack_target`）是基于推测的。**实现前必须检查 [scripts/systems_game/wave_manager.gd](../../scripts/systems_game/wave_manager.gd) 和现有 map_config 的 waves 格式**，以实际字段名为准。

### 风险2：.tres 文件手写易出错
建议在 Godot 编辑器中创建 MapConfig 资源，而非手写 .tres 文件。流程：
1. 编辑器中 → FileSystem → 右键 resources/ → New Resource → 选 MapConfig
2. 在 Inspector 中填写所有字段
3. 保存为 map17_config.tres

### 风险3：.tscn 场景结构复杂
建议复制现有 `map_1.tscn` 作为模板，然后修改：
1. 更换 map_config 引用
2. 调整 CapturePoint 节点位置和参数
3. 添加/移除 WaveManager 节点
4. 更换 VictoryCondition 节点
5. 确保 NavigationRegion2D 覆盖新地图范围

### 风险4：关卡解锁顺序
现有系统是线性解锁（通关 N 解锁 N+1）。盟友关卡系列（map_17~20）是接在 map_16 之后，还是作为独立分支？如果是独立分支，需要修改 `save_manager.gd` 的解锁逻辑。**建议先接在 map_16 之后（最简单），后续可调整。**

### 风险5：50+ 单位性能
关卡3最终会有 50+ 个 CharacterBody2D + AllyAI。AllyAI 已有 6 帧节流优化（`SCAN_THROTTLE_FRAMES=6`），但仍需实际测试帧率。如果卡顿，考虑：
- 增大 `SCAN_THROTTLE_FRAMES`（从 6 改到 8 或 10）
- 减少 AI 队友总数
- 减少敌方单位数
