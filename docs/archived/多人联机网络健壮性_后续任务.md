## 当前状态

Step 1（P0 网络健壮性）已完成并合并到 master。多人联机现在具备基本的网络异常反馈机制：

| 能力 | 现状 |
|---|---|
| 应用层心跳 | ✅ 客户端 1s 发 `app_ping`，服务器回 `app_pong` |
| 半开连接检测 | ✅ 游戏中连续 5s 无消息 → emit `connection_stale` |
| 服务器超时检测 | ✅ ws 上 5s 无 `_lastActivity` → 通知对端 + terminate |
| 掉线立即暂停 | ✅ LockstepSync 监听 `connection_lost` → `TickManager.pause()` |
| 网络状态 UI | ✅ 左上角状态条（绿/黄/红 + ping 值） + 错误对话框（4 种原因） |
| 主动/被动断开区分 | ✅ `_manual_disconnect` flag，避免主菜单退出误触发弹窗 |
| 对方关游戏即时反馈 | ✅ `player_left` → `connection_lost("peer_left")`，无需等心跳超时 |

测试通过的 3 个场景：
1. **正常对局**：双端 hash 一致，无 desync，状态条显示 ping
2. **对方关游戏**：客户端 2 关闭 → 客户端 1 立刻弹窗 + 暂停（无需等 5s）
3. **重进房间浏览器**：修复了 `ERR_ALREADY_IN_USE`（connect 幂等化）

## 已知限制

### L1. 心跳依赖服务器部署
服务器必须部署新版 `index.js` + `room_manager.js`，否则 `app_ping` 落到 `default` 分支不会回 `app_pong`。已通过加 `if not TickManager.enabled: return` 让客户端**在游戏中**仍能靠 game_data 保持 stale 计时器，但主菜单/lobby 阶段若服务器忘更新会丢失 ping 显示。

### L2. 暂停后无法重连
对方掉线 → 主机端暂停 + 弹窗 → 只有"返回主菜单"和"退出游戏"两个选项。没有"等待对方重连"或"我重连回去"的能力。lockstep 状态一旦分叉（一端继续推进 tick）就不可恢复，所以 Step 1 选择"直接结束对局"作为最简单可行的策略。

### L3. DESYNC 检测形同虚设
`StateHasher._compute_hash()` 只算 RNG seed/state（`hash_combine(h, TickManager.rng.seed)` + `rng.state`），**不算单位位置/血量/资源**。即使两端状态完全分叉，只要 RNG 一致就检测不到。`compute_full_hash()` 写了但没人调用。

### L4. 服务器无身份校验
`room_manager.relayGameData` 直接转发 payload，不校验 `payload.player` 是否匹配 ws 在房间里的真实身份。任何客户端可发 `{"player":2, ...}` 伪造对方命令。

### L5. DESYNC 仅 push_warning
即使 hash 比对失败，当前只 `push_warning("DESYNC ...")`，游戏继续跑，玩家无感知。

### L6. 命令迟到未处理
`CommandBuffer.receive_remote_commands` 不校验 tick 是否过期。如果命令迟到（应 tick=100 的命令 tick=105 才到），仍会执行，可能导致状态分叉。

### L7. ws 心跳 ping 阈值未触发状态条变色
`NetworkStatus._on_ping_updated` 中 100/250ms 阈值切换颜色，但实际网络正常的 ping 通常 < 50ms，黄色/红色分支几乎不会触发。可接受（玩家看到绿就够）。

## 后续任务

### Step 2（P0 延续）— 立刻可做，无需服务器再部署的部分

> 注：服务器身份校验（T2.2）会改 `server/room_manager.js`，需要再次部署。

#### T2.1. StateHasher 修 desync 检测
**目标**：让 desync 检测真正起作用，发现状态分叉时弹窗 + 暂停。

实现要点：
- `scripts/network/state_hasher.gd._compute_hash()` 改用现有的 `compute_full_hash(units_node, buildings_node, gold)`，包含实际游戏内容
- 由 `LockstepSync.set_game(self)` 注入 game 引用，`StateHasher.set_game(game)` 接收
- `_local_hashes` / `_remote_hashes` 用 Dictionary 缓存最近 10 个 tick 的 hash，处理"远端先到 vs 本地先到"的时序
- desync 检测到后：触发 `RelayManager.connection_lost.emit("desync")` → 复用现有的暂停 + 弹窗机制
- 弹窗文案已在 `NetworkStatus.REASON_TEXT["desync"]` 中准备好

测试：临时改 `_compute_hash` 让本地 hash 偏 1（`h += 1`），双端 hash 必然不一致，第一次 60 tick 比对（约 2 秒）时应弹 DESYNC 对话框。

#### T2.2. 服务器身份校验
**目标**：防伪造对方命令。

实现要点（`server/room_manager.js`）：
- 在 `relayGameData(ws, payload)` 入口加校验：
  ```js
  const claimedPlayer = payload && payload.player;
  const actualPlayer = this.getPlayerRole(ws);  // 已有
  if (actualPlayer === 0) return;
  if (claimedPlayer !== actualPlayer) {
    ws.send(JSON.stringify({ type: "error", payload: { message: "Player identity mismatch" } }));
    this.leaveRoom(ws);
    ws.terminate();
    return;
  }
  ```
- `loadComplete` 等其他游戏阶段方法复用 `actualPlayer` 判定
- 客户端不需要改：服务器踢人后 ws 关闭 → 触发 `disconnected` → 现有 `connection_lost("disconnected")` 路径

测试：用 wscat 伪装成 host，发 `{"type":"game_data","payload":{"player":2,...}}`，服务器应 `[security]` 日志 + terminate，正常玩家收到 `error` 不执行伪造命令。

#### T2.3. CommandBuffer 命令迟到处理（轻量）
**目标**：命令迟到时不要静默执行。

实现要点（`scripts/network/command_buffer.gd.receive_remote_commands`）：
- 如果 `tick_num < TickManager.tick_count - 5`（明显过期）→ `push_warning` + 跳过
- 如果 `tick_num < TickManager.tick_count`（轻微迟到）→ `push_warning` 但仍执行（避免卡死）
- 正常范围（`tick_num >= TickManager.tick_count`）→ 不变

### P1 — 重连 / Grace period（更大的工程）

#### T3. Grace period 重连
**目标**：玩家短时间掉线（30-120s）能在窗口内重连回内存中的对局，无需重传状态。

实现要点（业界标准做法，参考星际2/AOE4）：
- 掉线触发后**不立即终止对局**，进入 `WAITING_RECONNECT` 状态
- 主机端 / 对方看到"对方玩家断开，等待重连... 30s"倒计时
- 重连窗口内游戏暂停（已有 `TickManager.pause`）
- 客户端重连后用相同的 ws session id 恢复，跳过 lobby 流程
- 超时则判负

依赖：服务器需要保留房间状态 + 客户端 session 持久化。改动较大。

#### T4. 慢速玩家（lagging peer）处理
**目标**：一端帧率低 / 网络慢时不要拖垮另一端。

实现要点：
- 检测：CommandBuffer 中 `_pending[tick]` 在 `tick_count - N` 之前未齐 → 判定 lagging
- UI：状态条显示"Player X is lagging" + 倒计时
- 选项 1（最简单）：等待，无限暂停
- 选项 2：等待超时后允许其他玩家投票踢人
- 选项 3：被踢玩家由 AI 接管（最大工程量）

### P2 — 锦上添花

#### T5. Desync 自动保存 replay
**目标**：desync 发生时自动保存双方命令流供 debug。

实现要点：
- `CommandBuffer` 全程记录所有本地/远端命令到一个 Array
- desync 触发时序列化到 `user://desync_<timestamp>.json`
- 文件包含：tick、双端 hash、命令流、种子
- 用于事后离线重现 desync

#### T6. 房间清理 / 序列化
**目标**：服务器进程重启后能恢复房间状态。

实现要点：
- `server/room_manager.js` 定期把 rooms Map 序列化到文件/Redis
- 进程启动时读取
- 房间空闲 5 分钟自动销毁（防泄漏）

#### T7. 投票踢人 / 中途加入观战
不属于 P0 网络健壮性范畴，列为长期可做项。

## 部署 checklist

每次改 `server/` 后：
1. 拷贝 `server/index.js` + `server/room_manager.js` 到远程服务器对应路径
2. 重启 node 进程（pm2 / systemctl / 手动）
3. 验证：`grep -n "app_ping" server/index.js` 应有输出
4. 客户端连接后服务器日志应有 `type=app_ping` 且**不再出现** `[warn] Unknown message type: app_ping`
