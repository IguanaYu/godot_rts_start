# 多人联机势力系统 — 现状与后续任务

> 起点 commit: `8fe8587`（feat: 多势力系统 — faction/player/alliance 三层解耦）
> 适用 worktree: `lockstep-multiplayer`

## 当前状态

已完成三层概念解耦，旧 `Team { PLAYER, ENEMY }` 字段保留由 `alliance_id` 派生，向后兼容。

| 层 | 字段 | 决定 | 取值 |
|---|---|---|---|
| 颜色 | `faction_color` | 视觉贴图目录 | `Faction.ColorId`：BLACK/BLUE/PURPLE/RED/YELLOW/NEUTRAL |
| 玩家 | `owner_id` | 控制权 + 金币归属 | player_id (1-4)；AI = -1 |
| 联盟 | `alliance_id` | 胜负共享 / 敌对关系 | 0=玩家方；1=敌方；预留 2/3 给 PvP |

控制权规则（`combat_controller._can_control`）：
- `alliance_id != 0` → 不能控
- 同 alliance + 同 slot → 共享控制（co-op）
- 同 alliance + 异 slot → 只控 `owner_id == my_id` 的单位
- 跨 alliance → 互为敌人

地图配置新结构（`map_config.gd`）：
```
alliances: Array[Dictionary]  # 空 → 走旧字段 fallback
# alliance: { id, is_ai, slots: [slot] }
# slot:     { color, start_pos, initial_gold, units, buildings }
enemy_scaling: { per_player_unit_mult, per_player_extra_groups }
```

修复的 bug：
- "造兵造出红色" — `mp_spawn_unit` 改用 `alliance_id` 而非 `player == my_id` 推断
- "host 看不到 guest 单位" — `_on_player_joined` 用 deterministic pid=2 而非 server 推送的内部 id
- "敌方兵营造蓝色兵" — 建筑造兵继承建筑势力字段
- "建筑造的兵能选不能控" — 分配 `net_id` 并注册到 lockstep
- "客户端延迟高" — 建筑产兵 timer 由 `TickManager.tick` 推进，两端同步

## 已知限制

### L1. Lobby 选势力 UI 缺失
玩家无法在房间内选 slot 或颜色。当前是 deterministic 默认分配：
- player_id 1 → alliance 0, slot 0, color BLACK（素材缺，fallback 蓝色）
- player_id 2 → alliance 0, slot 1, color BLUE

无法测试同槽 co-op、异色盟友、玩家自选势力。

### L2. Relay server 只支持 2 玩家
`scripts/network/relay_manager.gd:76` `my_player_id = 2` 写死。server 当前协议下所有 client 的 my_id 都是 2，多 client 加入会冲突。

客户端代码（`network_manager.gd`、`command_buffer.gd`）已支持 N 玩家，瓶颈在 server。

### L3. 颜色素材缺失
`assets/units/` 和 `assets/buildings/` 只有 `blue_*` 和 `red_*`。缺：
- `assets/units/{black,purple,yellow}_{warrior,archer,lancer,monk}/`（12 个目录）
- `assets/buildings/{black,purple,yellow}_{house,tower,castle,barracks,monastery,archery}/`（18 个目录）

`unit.gd:_setup_texture` 和 `building.gd:_setup_texture` 有 fallback 到蓝色，所以功能正常但视觉上看不到真正多色。

### L4. 同槽 co-op 未实测
默认分配是异槽（slot 0/1），无法验证"同 alliance+slot 共享控制"。需要 L1 的 lobby UI 实现后才能测。

### L5. 金币独立池未充分验证
代码层 `main.gold` 仍是单一字段，per-player gold 通过 `NetworkManager.player_sessions[pid].gold` 维护。`_spawn_dynamic_players` 末尾把 my session 的 gold 复制到 `main.gold`，但实际游戏中造兵/建造扣 gold 是否真正 per-player 独立，需要观察。

### L6. 建筑 timer 部分同步
只有 `_production_process`（产兵）改用 `TickManager.tick`。其他 timer 仍用 `_process(delta)`：
- `_tower_process`（塔攻击）
- `_aura_process`（光环）
- `_wall_repair_process`（墙修复）

如果两端帧率差异大，这些状态计算会不一致（伤害/治疗量不同步）。

## 后续任务

### P0 — 核心功能补全

#### T1. Lobby 选势力 UI
**目标**：让玩家在 lobby 选 alliance、slot、颜色。

实现要点：
- `scripts/ui/lobby.gd` 显示当前地图 `map_config.alliances[0].slots` 列表
- 每个 slot 显示：颜色色块、占用状态（空 / Player N）
- 玩家点 slot 占位 → 调 `NetworkManager.set_player_slot(my_id, alliance, slot, color)`
- host 维护权威 `slot_occupation`，通过新 relay 消息类型 `slot_select` / `slot_update` 同步给所有 client
- 已被占用的 slot 显示"加入共享"按钮（同槽 co-op）

依赖：T2（同步协议）

#### T2. Relay slot 同步协议
**目标**：host 把 slot 选择同步给所有 client，保证两端 `player_sessions` 一致。

实现要点：
- `relay_manager.gd` 加 `send_slot_select(alliance, slot, color)` 和信号 `slot_updated(player_id, alliance, slot, color)`
- server 透传（已有 game_data 通道，加新 type 即可）
- `network_manager.gd` 收到 slot_updated 时更新对应 player 的 session

#### T3. Relay server 4 人支持
**目标**：突破 2 人限制。

实现要点（server 端，仓库路径 `server/`）：
- server 在 `room_joined` payload 包含 `player_id`（递增分配，不再固定 2）
- `relay_manager.gd:76` 改为 `my_player_id = payload.get("player_id", 2)`
- `network_manager.gd:_on_player_joined` 不再用 deterministic pid=2，改用 server 推送的真实 pid

### P1 — 体验完善

#### T4. 补齐颜色素材
**目标**：5 色全部可见。

需要补：12 个 unit 目录 + 18 个 building 目录（见 L3）。补完后 fallback 自动停用。

#### T5. 金币真正独立
**目标**：每个 player 的 gold 完全独立，即使同槽 co-op 也不共享。

实现要点：
- `main.gd` 移除 `gold` 单一字段，改读 `NetworkManager.player_sessions[my_id].gold`
- 造兵/建造扣 gold 时只扣 my_id 的（命令发起方付钱）
- UI 显示 my_id 的 gold
- 同槽 co-op 时：单位共享控制权，但谁点造兵谁付钱

#### T6. 同槽 co-op 验证
依赖 T1。测试场景：
- 两玩家都选 alliance 0 slot 0
- 验证：单位颜色相同、控制权共享（任一玩家可指挥所有同 slot 单位）、金币各自扣

### P2 — 扩展性

#### T7. N 联盟 PvP 测试
**目标**：验证代码层 N 联盟扩展性。

实现要点：
- 加 PvP 测试地图（alliance 2 个，每个 1-2 slot）
- 验证跨 alliance 玩家互为敌人
- 验证胜负条件按 alliance 判定（任一 alliance 全灭则另一方胜）

#### T8. 敌方随玩家数动态增强
**目标**：玩家越多敌人越强。

`MapConfig.enemy_scaling` 已定义未实现：
```
{ "per_player_unit_mult": 1.0, "per_player_extra_groups": [] }
```

在 `_spawn_ai_alliances_from_config` 按 `NetworkManager.player_count` 缩放 unit 数；或追加 extra_groups 波次。

#### T9. 建筑 timer 全部 tick 同步
**目标**：消除两端帧率差异导致的状态不一致。

把 `_tower_process` / `_aura_process` / `_wall_repair_process` 也改用 `TickManager.tick` 推进（参考 `_on_production_tick` 实现）。

### P3 — 协议层加固

#### T10. lockstep 断线重连
**目标**：一端断开时另一端不卡死。

加心跳检测 + 重连逻辑。当前断开后 `command_buffer._check_ready` 永远等不到对方命令。

#### T11. StateHasher 分歧处理
**目标**：hash 不匹配时报警/回滚。

`StateHasher` 已有 hash 计算和对比，但发现分歧时只打 log，不暂停或回滚。加分歧处理 UI（"不同步，是否重开"）。
