## 0. 设计目标

| 对战场景 | 目标 TTK | 设计意图 |
|---------|---------|---------|
| T1 vs T1 | **10s** | 基准节奏，有操作空间 |
| T2 vs T1 | 6s | 高级兵秒低级兵 |
| T1 vs T2 | 18s | 低级兵打高级兵很慢 |
| T3 vs T1 | 3s | 终极兵碾压 |
| T1 vs T3 | 30s | 几乎打不动 |
| T2 vs T2 | 12s | 中阶兵互殴 |
| T3 vs T3 | 15s | 重甲对决 |

## 1. 核心公式选择

**减法公式（推荐）：**
```
实际伤害 = max(攻击力 - 护甲, 1)
DPS = 实际伤害 / 攻击间隔
TTK = HP / DPS
```

**为什么选减法而非百分比：**
- 计算简单，玩家能在脑中预算
- 通过护甲上限（0-5）避免无限刮痧
- 最小伤害=1 防止完全打不动

## 4. 推荐采用方案 A 的理由

**对比三方案的关键指标：**

| 指标 | 方案 A | 方案 B | 方案 C |
|------|-------|-------|-------|
| T1 vs T1 TTK | 8s | 11s | 6s |
| 符合 10s 目标 | 接近 | **最符合** | 较远 |
| T3 vs T1 TTK | 4.4s | 5.3s | 3.3s |
| 击中数区间 | 5-32 | 6-50 | 4-23 |
| 数值复杂度 | 中等 | 高（多护甲） | 低 |
| 操作空间 | **好** | 太慢 | 紧张但浅 |

**为什么 A 优于 B：**
- B 方案的 T1 打 T3 要 50 刀，玩家体验差
- B 方案数值偏大，调试成本高
- A 方案 T1 打 T3 也要 32 刀，但整体节奏更紧凑

**为什么 A 优于 C：**
- C 方案 TTK 太短，玩家来不及反应
- 6 秒 TTK 接近 SC2 节奏，但失去了"10 秒目标"的初衷

## 5. 引入克制关系（在方案 A 基础上）

**3×3 攻击类型 × 护甲类型倍率表：**

| 攻击\护甲 | 轻甲 | 中甲 | 重甲 |
|----------|------|------|------|
| **普通** | 1.00 | 1.00 | 1.00 |
| **穿刺** | **1.50** | 1.00 | 0.75 |
| **魔法** | 0.75 | 1.00 | **1.50** |

**应用后 TTK 变化（T1 标准兵 HP=80, DMG=10, CD=1.0）：**

| 攻击方 | 防御方 | 倍率 | 实际 DPS | TTK |
|--------|--------|------|---------|-----|
| 穿刺打轻甲 | 轻甲 | 1.5x | 15.0 | **5.3s** |
| 普通打中甲 | 中甲 | 1.0x | 10.0 | 8.0s |
| 魔法打重甲 | 重甲 | 1.5x | 15.0 | **5.3s** |
| 穿刺打重甲 | 重甲 | 0.75x | 7.5 | **10.7s** |
| 魔法打轻甲 | 轻甲 | 0.75x | 7.5 | **10.7s** |

**克制效果：** 克制时 TTK 从 8s 降到 5.3s（-34%），被克制时升到 10.7s（+34%）。这个幅度比 WC3 的 200% 倍率温和，但足以让玩家有"换兵种应对"的动机。

## 6. 经济匹配验证

**关键问题：** 10 秒 TTK 内，玩家能补充多少兵力？

```
假设单矿收入：500 资源/分 ≈ 8.3 资源/秒
假设 T1 兵造价：50 资源，生产时间 15s

TTK = 10s → 单矿 10s 内产出 83 资源 ≈ 1.7 个 T1 兵
```

**结论：**
- TTK 10s 时，**战中补充几乎无效**（1.7 个兵改变不了战局）
- 战斗胜负 = 已有兵力 + 阵型 + 微操
- 这符合大多数 RTS 的设计哲学：**战前决策 > 战中操作**

**如果想战中补充有效：**
- TTK 拉到 20s+（WC3 方向）
- 或降低造价到 25 资源（SC2 Zergling 方向）
- 或缩短生产时间到 8s（爆兵流派）

## 7. 不同兵种角色的数值差异化

基于 T1 = 80HP / 10DMG / 1.0CD 基准，各角色推荐数值：

| 兵种类型 | HP | 伤害 | CD | 护甲 | DPS | 自打 TTK | 击中数 |
|---------|----|------|----|------|------|--------|-------|
| 近战轻甲兵 | 80 | 10 | 1.0 | 0 | 10.0 | 8.0s | 8 |
| 近战重甲兵 | 100 | 8 | 1.0 | 2 | 6.0 | 16.7s | 17 |
| 远程脆皮兵 | 60 | 12 | 1.0 | 0 | 12.0 | 5.0s | 5 |
| 远程标准兵 | 75 | 10 | 1.1 | 0 | 9.1 | 8.3s | 8 |
| 近战肉盾兵 | 120 | 8 | 1.2 | 2 | 5.0 | 24.0s | 20 |
| 刺客型兵 | 65 | 14 | 1.0 | 0 | 14.0 | 4.6s | 5 |

**设计规律：**
- **肉盾**：HP 高 + 护甲高 + DPS 低 → 拉长 TTK
- **刺客**：HP 低 + 伤害高 + DPS 高 → 极短 TTK
- **重甲**：HP 中 + 护甲高 + DPS 低 → 难杀也难杀人
- **脆皮**：HP 低 + DPS 高 → 高风险高回报

## 8. 升级系统的数值影响

**攻击升级（+1/+2/+3 伤害）：**

| 升级等级 | T1 伤害 | 对 0 甲 DPS | 对 0 甲 TTK | 对 2 甲 DPS | 对 2 甲 TTK |
|---------|--------|------------|------------|------------|------------|
| 0 | 10 | 10.0 | 8.0s | 8.0 | 10.0s |
| +1 | 11 | 11.0 | 7.3s | 9.0 | 8.9s |
| +2 | 12 | 12.0 | 6.7s | 10.0 | 8.0s |
| +3 | 13 | 13.0 | 6.2s | 11.0 | 7.3s |

**护甲升级（+1/+2/+3 护甲）：**

| 升级等级 | T1 护甲 | vs 10 伤 DPS | vs 10 伤 TTK | 边际收益 |
|---------|--------|-------------|-------------|---------|
| 0 | 0 | 10.0 | 8.0s | — |
| +1 | 1 | 9.0 | 8.9s | +11% 生存 |
| +2 | 2 | 8.0 | 10.0s | +25% 生存 |
| +3 | 3 | 7.0 | 11.4s | +43% 生存 |

**关键发现：** 减法公式下，护甲升级的边际收益**递增**（每多点护甲收益更大）。这与 WC3 百分比公式相反。

**含义：**
- 攻击升级在前期更划算（线性收益）
- 护甲升级在后期更划算（指数收益）
- 这创造了天然的"攻防博弈"——何时升级哪个？

## 9. 总结：可直接套用的数值表

```python
# 项目可用的基准数值表（基于方案 A）

T1_BASE = {
    "hp": 80,          # 基础 HP
    "damage": 10,      # 基础伤害
    "cooldown": 1.0,   # 攻击间隔（秒）
    "armor": 0,        # 基础护甲
    "cost": 50,        # 造价
    "build_time": 15,  # 生产时间（秒）
    "supply": 1,       # 人口
}

T2_BASE = {
    "hp": 150,         # T1 的 1.8x
    "damage": 14,      # T1 的 1.4x
    "cooldown": 1.0,
    "armor": 1,        # +1
    "cost": 125,
    "build_time": 30,
    "supply": 2,
}

T3_BASE = {
    "hp": 250,         # T1 的 3.0x
    "damage": 18,      # T1 的 1.8x
    "cooldown": 1.0,
    "armor": 2,        # +2
    "cost": 250,
    "build_time": 45,
    "supply": 4,
}

# 攻击类型 × 护甲类型倍率
DAMAGE_MULTIPLIER = {
    "普通": {"轻甲": 1.0, "中甲": 1.0, "重甲": 1.0},
    "穿刺": {"轻甲": 1.5, "中甲": 1.0, "重甲": 0.75},
    "魔法": {"轻甲": 0.75, "中甲": 1.0, "重甲": 1.5},
}

# 伤害公式
def compute_damage(attack, defender_armor, atk_type, def_armor_type):
    base = max(attack - defender_armor, 1)
    return base * DAMAGE_MULTIPLIER[atk_type][def_armor_type]

# TTK 公式
def compute_ttk(attacker, defender):
    dmg_per_hit = compute_damage(
        attacker["damage"], defender["armor"],
        attacker.get("atk_type", "普通"),
        defender.get("armor_type", "中甲")
    )
    dps = dmg_per_hit / attacker["cooldown"]
    return defender["hp"] / dps if dps > 0 else float('inf')
```

## 10. 下一步可做的验证

1. **A/B 测试** — 在项目里实现方案 A，实际玩 5-10 局，记录主观节奏感
2. **微调 Tier 跨度** — 如果 T2 vs T1 太快/太慢，调整 T2 的 HP 或 DMG 倍率
3. **引入射程** — 远程兵应该有射程优势，可通过"实际输出时间"折算到 DPS
4. **引入 AOE** — 溅射单位的有效 DPS 应乘以"预期命中目标数"（通常 1.5-2.5）
5. **引入技能** — 技能伤害应按"等效 DPS"（伤害/CD）折算到单位总输出

---

> **核心原则：** 数值设计不是一次成型，而是"基准值 + 反复迭代"。先用这套数值实现 MVP，再根据实际体验微调。
