# 资源面板 UI — 跨游戏调研

> 数据源: 星际2 / 魔兽3 官方文档 + Liquipedia + Battle.net 经典站 + Reddit/Dota2 Wiki/Hive Workshop/TeamLiquid
> 采集日期: 2026-07-07
> 用途: 为本项目的"模块化资源信息面板"改造提供决策支撑
> 调研重点: ① 金币变化怎么显示 ② 人口怎么显示 ③ 击杀/统计等扩展信息怎么组织 ④ 模块化的设计模式

---

## 0. 一句话总结

资源面板没有"标准答案"，但有清晰的**两类范式**——RTS 范式（星际/魔兽/AoE，右上角静态条，强调总量和警示）和 MOBA 范式（Dota/LoL，顶部计分板 + 底部个人 HUD，强调每次获得的反馈）。本项目偏 RTS，**骨架走星际/魔兽路线，反馈机制可借鉴 MOBA 的 +X 浮动**。模块化的核心是"资源项注册机制"，不是 UI 排版。

## 2. 案例事实

### 2.1 资源面板的位置与朝向

| 游戏 | 位置 | 朝向 | 备注 |
|------|------|------|------|
| SC2 | 右上角 | 横向 3 组 | 水晶/气体/补给 |
| WC3 | 右上角 | 纵向 3 行 | 金币/木材/食物 |
| AoE2/4 | 左上角 | 横向 4-5 组 | 食物/木材/黄金/石头/(村民) |
| Dota2 | 屏幕正上方（计分板） | 横向 | 击杀+时长，金币在底部 HUD |
| LoL | 屏幕正上方（计分板） | 横向 | 击杀+KDA，金币在底部 HUD |

**结论**：
- **RTS 普遍在角落**（右上是 Blizzard 派、左上是 Ensemble 派），因为 RTS 同时操作大量单位，资源要常驻但不挡视野
- **MOBA 普遍在正上方**（计分板），因为 MOBA 视野跟随英雄，且双方对抗的核心信息（比分、时长）必须平等可见
- 本项目是 RTS，**保持现有左上角或迁到右上角都合理**，主要看主战场视野留白

### 2.2 金币 / 资源本身的显示格式

| 游戏 | 显示内容 | 字体/颜色 |
|------|---------|----------|
| SC2 水晶 | `1234`（纯数字，无千分位） | 白色 |
| SC2 气体 | `456` | 绿色（与水晶不同色调） |
| SC2 补给 | `24/50`（已用/上限） | 白色，警戒时变黄/红 |
| WC3 金币 | `1500` | 金色字体 |
| WC3 木材 | `200` | 浅棕色 |
| WC3 食物 | `30/50`（已用/上限） | 灰白，Upkeep 触发变色 |
| AoE2 黄金 | `500` | 金色 |
| Dota2 金币（玩家本人） | `2345`（带千分位） | 金色，下方显示 GPM |

**关键观察**：
- **几乎所有 RTS 都不用千分位**（`1500` 而非 `1,500`），因为数字一般不超过 5 位
- **MOBA 用千分位**（`2,345`），因为后期金币能上 5-6 位
- 本项目初始金币 `10000`、可累积到几万，**建议带千分位**（MOBA 风格）以便快速识别
- 显示格式是「图标 + 数字」紧凑组合，**不是「金币: 1234」这种带冒号标签的写法**——图标本身就是标签

### 2.3 "加多少钱"——金币变化的反馈机制

这是用户最关心的点。三种主流方案：

#### 方案 A：数字本身短暂变色（星际/魔兽派）

- **SC2**：矿车返回投放时，水晶数字**短暂泛绿**约 0.3s，无 +X 文字
- **WC3**：金币/木材数字变化时，**本身短暂变绿（获得）或变红（消耗）**
- **优点**：不刷屏、不打扰、信息密度高
- **缺点**：单次变化金额不直观，玩家只知道"变了"不知道"变多少"
- **适用场景**：高频小额变化（采集、维护扣减）

#### 方案 B：浮动 +X 文字（MOBA 派）

- **Dota2**：击杀英雄/小兵、拆建筑时，**屏幕上方出现绿色 +XX 浮动文字**，向上漂浮约 1s 淡出
- **LoL**：类似机制，金币、经验都有 +X 浮动
- **优点**：反馈强烈、激励感强、单次金额清晰
- **缺点**：高频时刷屏（Dota2 客户端有过滤器，连续相同来源的浮动会合并）
- **适用场景**：低频大额事件（击杀、奖励、任务）

#### 方案 C：图标变化提示（魔兽3 Upkeep 派）

- **WC3 Upkeep 机制**：食物超过 30/60/80 时，金币图标本身**从金色 → 银色 → 红色**变化，提示"现在采矿效率降低"
- 这是"间接反馈"——不是告诉你"你赚了多少"，而是"你当前的赚钱状态如何"

#### SC2 的特殊设计：补给被摧毁时的红字

- 当补给站被摧毁导致**已用 > 上限**时，补给数字 `27/24` 整体变红，且屏幕中央会有 "Supply Capped" 警告
- 这是 RTS 独有的设计——**资源上限被动态破坏**比静态上限更紧张

**结论建议**：
- **采集类（被动收入）**：用方案 A（变色），不要 +X，否则刷屏
- **事件类（击杀/奖励）**：用方案 B（+X 浮动），让玩家有获得感
- **状态类（人口/upkeep）**：用方案 C（图标变色），传达"当前状态"
- **本项目已有 [floating_text.gd](../../scripts/effects/floating_text.gd) 工具**，方案 B 实现成本极低

### 2.4 人口 / 补给的显示

| 游戏 | 格式 | 警戒色机制 |
|------|------|-----------|
| SC2 | `24/50` | 已用 ≥ 上限 80% → 黄；超上限 → 红 + 屏幕警告 |
| WC3 | `30/50` | Upkeep 阈值（30/60/80）触发金币图标变色 |
| AoE2 | 不显示人口（默认）； AoE4 显示 `已用/上限` | 接近上限变红 |
| Dota2 | 无人口概念 | — |
| LoL | 无人口概念 | — |

**RTS 共性**：
- **必用 `已用 / 上限` 双数字**，没有只显示单边的
- **至少两档警戒**：接近上限（黄）+ 超上限（红）
- WC3 的 Upkeep 是**用资源图标间接传达人口压力**的巧妙设计——人口超阈值不直接惩罚人口，而是惩罚金币效率

**本项目建议**：
- 格式：`18/50` 双数字
- 上限来源：`map_config` 里定义（如果有），否则用一个全局常量
- 警戒：≥80% 黄、≥100% 红、>100% 闪烁（如果允许超）
- 算法：count(player_units) - count(free_building) / pop_cap

### 2.5 战斗统计（击杀/部队数/计时）—— MOBA 的做法

MOBA 顶部计分板是 RTS 没有的设计，因为 MOBA 关注"双方对抗的宏观态势"：

| 信息 | Dota2 位置 | LoL 位置 |
|------|----------|----------|
| 双方击杀比 | 屏幕正上方，超大数字 `12 : 8` | 同 Dota2 |
| 比赛时长 | 击杀数下方，`23:47` | 同 Dota2 |
| 玩家本人 K/D/A | 底部 HUD | 底部 HUD |
| 玩家本人金币/GPM | 底部 HUD | 底部 HUD |
| 净资产（Net Worth） | 仅观战/录像可见 | 仅观战/录像可见 |
| 队友/敌人装备 | 计分板点击展开 | 计分板点击展开 |

**RTS 借鉴点**：
- RTS 战役/合作模式里，**击杀数、损失数、波次数、生存时间**是常见的"事后复盘"指标
- SC2 战役结算页会显示：用时、击杀、最佳单位、APM 等——**这些不在战斗中显示，而在结算时**
- WC3 战役同样：战中只看资源，结算看战损统计
- **例外**：SC2 合作模式的指挥官等级条会战中显示（右上角），因为它是"长期进度"而非"本局指标"

**本项目建议**：
- **战中（默认显示）**：金币、人口、科技、升级币 — 已经在做的
- **战中（可选开关）**：击杀数、损失数、生存时间、波次 — 给硬核玩家
- **战后结算**：完整统计 — 已经有的星级评价系统可以扩展

### 2.6 模块化设计模式（架构层面）

这是用户最关心的"未来想加什么就加什么"。

#### 业界共识：REA 模式 + 组件化

学术界对 RTS 资源系统的形式化叫 **Resource Entity Action (REA) 模式**（参考 Spronck 论文）。落到 UI 层面，社区共识是：

```
面板（容器） —— 只管布局和刷新
  ├── 资源项 A（节点）—— 自带数据源 + 显示逻辑
  ├── 资源项 B（节点）
  └── 资源项 C（节点）
```

每个资源项是一个**自包含的节点**，面板不关心项的具体内容，只关心"你注册了、我就给你一行位置"。

#### 具体接口设计（综合社区方案 + Godot 实践）

```gdscript
# 注册时传一个配置字典
ui.resource_panel.register("kills", {
    icon = preload("res://ui/icons/sword.png"),
    color = Color.RED,
    getter = func(): return Stats.kills,         # 取值闭包
    formatter = func(v): return str(v),          # 可选格式化
    on_change = func(old, new):                  # 可选变化回调
        if new > old:
            FloatingText.show("+%d" % (new - old), color=Color.GREEN),
    group = "combat",                              # 可选分组
    visible_by_default = true,                     # 是否默认显示
})
```

#### 关键设计原则（社区试错出来的）

1. **数据驱动**：getter 是闭包，UI 不持有数据，数据源在哪都行
2. **节流刷新**：每帧调 getter，但只在值变化时更新 Label（避免 60fps 刷新几十个 Label 掉帧）
3. **分组可选**：资源项可分组（"核心"/"战斗"/"时间"），分组支持开关
4. **动态增删**：register/unregister 是 O(1)，新加项时插入末尾或按 group 排序
5. **配置持久化**：玩家选择显示哪些项的设置保存到 settings

#### 反例：硬编码 Label 的痛点（本项目现状）

参考 [scripts/systems/game_ui.gd:413-503](../../scripts/systems/game_ui.gd#L413-L503)：
- 4 个独立 Label 变量 + 4 个独立 update 函数
- 加新指标要改 3 处（声明、布局、update 函数）
- 容易漏改、容易命名冲突、容易忘记 connect 信号

## 3. 试错教训（社区与设计师改过的坑）

1. **采集类资源用 +X 浮动会刷屏**
   - SC2 早期 Beta 试过给采矿加 +X 浮动，结果地图上漂浮数字铺天盖地，玩家根本看不清战斗
   - 后来改成"数字本身短暂变色"，问题立刻解决
   - **教训**：**只有玩家主动行为产生的收入（击杀、任务、奖励）才用 +X**，被动收入（采集、自动产出）用变色

2. **资源项过多会读不过来**
   - SC2 社区 compact UI 概念帖（TeamLiquid）证明：超过 5 项横向排列时，玩家瞄一眼无法识别
   - AoE 系列的解决方案是**重要项大字 + 次要项小字**的视觉层级
   - **教训**：模块化不等于平铺——必须有**视觉权重分级**

3. **每帧 _process 刷新所有 Label 会掉帧**
   - Reddit r/gamedev 多个帖子报告：RTS 后期单位多时，UI 刷新占用 5-10% CPU
   - **教训**：**dirty 标记 + 节流**——只在值变化时更新 Label，每帧只调 getter 不调 Label.set_text

4. **资源面板动态变化会破坏肌肉记忆**
   - Dota2 HUD 改版时玩家强烈抗议"我的金币栏怎么换位置了"
   - **教训**：注册项一旦注册，**位置应稳定**（用 group 内顺序保证），不要每次按字母排序

5. **千分位分隔符在 RTS 里反而碍事**
   - WC3/SC2 都不用千分位，因为 RTS 数字节奏快，玩家瞄一眼 `12345` 比读 `12,345` 更快
   - 但 MOBA 节奏慢、数字大，千分位有助读
   - **教训**：本项目数字规模看具体数值——金币初始 `10000`、可累积，**建议用千分位**；但击杀/人口等小数字不需要

6. **绝对定位 vs 容器布局**
   - 项目当前 [game_ui.gd:413-503](../../scripts/systems/game_ui.gd#L413-L503) 用的是 PanelContainer + VBoxContainer，已经接近正确做法
   - **不要**退回到绝对坐标（offset_left=6, offset_top=6 写死），否则分辨率适配会崩

## 4. 新增方向（用户没问但相关的）

### 4.1 资源面板之外——mini 面板（屏幕中央提示）

参考 SC2 的 "Supply Capped" / WC3 的 "Upkeep!" —— 某些**状态级**信息（人口超限、金币耗尽、波次开始）应该用屏幕中央的瞬态提示，而不是塞进资源面板。

本项目已有 [floating_text.gd](../../scripts/effects/floating_text.gd)，可以做"屏幕中央 1.5s 大字警告"。

### 4.2 资源历史曲线（高级功能）

参考 Dota2 Plus 的"实时 GPM/XPM 曲线"——把资源信息**可视化成时间序列**，让玩家看到自己的经济节奏。

本项目如果要加，可以用一个可折叠的"统计面板"（Tab 键打开），里面放曲线图。但**这超出了用户当前需求**，仅作为未来可能性记录。

### 4.3 资源项的图标 vs 字标

- SC2/WC3：用**真实图标**（晶体堆、气体罐、金币堆）
- AoE2：用**简笔图标**
- Dota2：金币用 `$` 字符 + 数字
- **本项目**：建议用简单图标（可以从 [Kenney.nl](https://kenney.nl/assets) 或 [Game-icons.net](https://game-icons.net/) 找免费资源），如果不愿意做美术，**用 emoji/Unicode 字符**（🪙/⚔️/🛡️）作为 MVP 方案也行——但 GDScript 字体要支持。

### 4.4 "信息分类"维度——按节奏分而非按主题分

与其按"经济/战斗/时间"分类，不如按**更新频率**分类：
- **高频项**（每帧/每秒变）：金币、人口 — 大字、显眼
- **中频项**（每几秒变）：科技等级、升级币 — 中字
- **低频项**（每几十秒/分钟变）：击杀数、波次、生存时间 — 小字、可折叠

这种分类法对模块化设计的影响是：**register 时传一个 priority/frequency 参数**，面板按 priority 自动排版。

### 4.5 战役/合作模式的"长期进度"

参考 SC2 合作模式的指挥官等级条——某些信息是**跨关卡**的（玩家等级、成就进度），不是本局指标。本项目如果未来有局外成长（看 [docs/brainstorming/局外成长设计调研.md](../brainstorming/局外成长设计调研.md)），这部分信息要**和本局资源分开**，不能混在一个面板里。

## 5. 待决策思考点（不替用户拍板）

1. **位置**：保持现有左上角（AoE 风格），还是迁到右上角（SC2/WC3 风格）？
   - 影响因素：现有 UI 已经在左上角、玩家已习惯；右上角更"正统"但需要重新布局

2. **金币反馈方式**：A / B / C（详见 2.3 节）选哪个，还是混合？
   - A：数字变色（不打扰）
   - B：+X 浮动（强反馈）
   - C：图标变色（状态指示）
   - 混合方案：采集用 A、击杀用 B、人口用 C

3. **千分位**：金币显示 `12345` 还是 `12,345`？击杀/人口等小数字要不要千分位？

4. **默认显示哪些项**？基于项目数据可用性，候选项有：
   - 核心：金币、人口、科技、升级币（已显示）
   - 战斗：击杀、损失、净击杀、活跃单位数
   - 时间：生存时间、当前波次、距下波倒计时
   - 性能：FPS（已有，可开关）
   - 问：战斗/时间类是默认显示、还是默认隐藏+按钮开关？

5. **模块化颗粒度**：
   - 轻量：只做"资源项注册"，UI 布局固定（一行或一列）
   - 中量：支持 group 分组 + 组级开关
   - 重量：完整可拖拽/可隐藏单项（玩家自定义 HUD）

6. **是否同时改造现有 4 个 label**？
   - A：新系统上线，老 4 个 label 一起改造（破而后立）
   - B：老接口保留兼容，新系统并行，逐步迁移
   - C：老接口废弃，一次性切换

7. **人口上限来源**：项目当前有人口系统吗？如果没有，**先用固定值（如 50/100）还是从 map_config 读？** 没有人口上限的话，"显示人口"也变成"显示当前单位数"（单数字）而非 `已用/上限`。

8. **击杀数据来源**：项目当前只有 `_units_lost`（[main.gd:73](../../scripts/main.gd#L73)，用于星级评价），没有正向击杀统计。要新增 `kills` 字段吗？记谁的击杀（玩家/AI/英雄）？
